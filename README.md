# ResumeRadar — AI Resume & ATS Match Analyzer

React + Vite + Tailwind frontend that extracts text from an uploaded PDF resume
client-side, sends it to a Vercel serverless function alongside a pasted job
description, and renders an ATS score, matching/missing skills, and rewrite
suggestions generated by the Gemini API.

## How it works (architecture)

1. **PDF → text, in the browser.** `pdfExtractor.js` uses `pdfjs-dist` to read
   every page of the uploaded PDF and reconstruct line breaks from text
   position data. Nothing is uploaded to a server at this step — only the
   extracted plain text leaves the browser, later, when you click *Run
   analysis*.
2. **Resume text + job description → `/api/analyze`.** This is a Vercel
   serverless function (Node runtime), not a client-side call to Google.
   Keeping it server-side means your `GEMINI_API_KEY` is never shipped in the
   JS bundle or visible in browser dev tools.
3. **`/api/analyze` → Gemini API.** The function builds a single prompt with
   the resume and job description, and calls `generateContent` on
   `gemini-2.5-flash` with a `responseSchema` so Gemini is constrained to return:
   `atsScore`, `scoreSummary`, `matchingSkills`, `missingSkills`, `suggestions`.
4. The function validates/clamps the parsed result (score forced into 0–100,
   arrays capped) before returning JSON to the client.

## 1. Prerequisites

- Node.js 18+
- A Gemini API key from [aistudio.google.com/apikey](https://aistudio.google.com/apikey)
- (Recommended for local dev) the Vercel CLI, so `/api` functions run locally
  exactly as they will in production: `npm i -g vercel`

## 2. Install

```bash
npm install
```

## 3. Environment variables

Copy the example file and add your key:

```bash
cp .env.example .env
```

```env
GEMINI_API_KEY=your_gemini_api_key_here
GEMINI_MODEL=gemini-2.5-flash   # optional override
```

`GEMINI_API_KEY` is read by `api/analyze.js` via `process.env` — it is **not**
prefixed with `VITE_`, so Vite will never bundle it into client code.

## 4. Run locally

The serverless function in `api/` only runs under Vercel's runtime, so for
full-stack local development use the Vercel CLI (it serves the Vite frontend
*and* `/api/*` together, reading `.env` automatically):

```bash
vercel dev
```

If you only need to work on UI/styling and don't need live Gemini calls, plain
Vite is faster:

```bash
npm run dev
```

(API calls will 404 in this mode since no serverless runtime is attached.)

## 5. Build

```bash
npm run build
npm run preview   # sanity-check the production build locally
```

## 6. Deploy to Vercel

```bash
npm i -g vercel    # if you haven't already
vercel              # first deploy — follow the prompts
vercel --prod       # promote to production
```

Then in the Vercel dashboard: **Project → Settings → Environment Variables**,
add `GEMINI_API_KEY` (and optionally `GEMINI_MODEL`) for the Production
(and Preview) environments, then redeploy. `vercel.json` already points Vercel
at the Vite build output and registers `api/analyze.js` as a function.

## Design notes

- Visual system: warm off-white surface, near-black ink, teal for matches,
  coral for gaps/errors — built around a "scanning" metaphor (the radial
  ATS gauge, the scan-line loading state) that matches what the tool
  actually does.
- Type: Space Grotesk (display), Inter (body), JetBrains Mono (scores,
  counters, data) — the mono face is reserved for numbers and status text to
  reinforce the "readout" feel.
- Fully responsive: single column on mobile, two-column input/results split
  from `lg:` breakpoint up. Focus-visible states and `prefers-reduced-motion`
  are respected throughout.

## Customizing the prompt / scoring logic

Edit `buildPrompt()` and `RESPONSE_SCHEMA` in `api/analyze.js` — for example
to weight certain sections more heavily, return additional fields (e.g. a
breakdown by category), or swap in a different Gemini model via
`GEMINI_MODEL`.
